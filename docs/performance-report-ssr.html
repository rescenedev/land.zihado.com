1. 전환 요약 — 빈 껍데기+클라 fetch → SSR seed

이전: 라우트가 빈 셸을 페인트 → 클라이언트 JS가 API fetch → 재렌더 (스켈레톤 깜빡임 + fetch 워터폴).
현재: 서버 컴포넌트가 워커 KV에서 데이터를 받아 HTML에 실데이터를 직접 seed → 첫 페인트에 차트·카드·거래목록이 즉시. 전 사이드바 메뉴가 실제 라우트(공유 가능 URL) + ISR 캐시.

검증: /stats HTML에 로딩 플레이스홀더 0개, RSC 페이로드(__next_f) + 통계 마커(중위·백분위 ×8) 서버 렌더 확인. 앱 콘솔 에러 0건.

5. 발견 + 조치 (배포 완료)

항목발견조치
PRERENDER 콜드 꼬리 데이터는 일1회 갱신인데 페이지 revalidate=120 → 2분마다 재생성. 경계 첫 방문자가 워커까지 내려가 /today 228ms·max 478ms. revalidate 120→1800s (7 라우트 + ssr.ts fetch). 재생성 15배↓. 워커 cron warmCore→warmSsrPages 추가 → 재생성을 cron이 흡수. 실효: 다음 cron 09:30 KST.
k6 테스트 부채 회귀 아님. buildUrl이 구 계약 ?lawdCd=&dealYmd= 호출 → 현행 ?dataset=®ion=&yyyymm=와 불일치로 스모크 50% 실패. config·params·smoke·load·stress·validation 6파일 현행 계약으로 정정. checks 210/210, http_req_failed 0%.

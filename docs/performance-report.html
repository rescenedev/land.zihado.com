1. 핵심 전환 — 병목은 빛 속도가 아니라 colo 라우팅

서버 SQL을 ~6ms까지 줄여도 50ms를 못 깬 진짜 원인은 네트워크 거리. 실측:

• Cloudflare 고객 zone: 같은 회선에서 도쿄(NRT) 37ms (Pro 업글해도 ICN 안 됨, BGP)
• Vercel: 같은 회선에서 서울(ICN) ~7ms (x-vercel-id: icn1)

→ 프론트+엣지캐시를 Vercel(ICN)로 이전, land.zihado.com을 DNS-only(회색 구름)로 직결. 데이터 원본(D1/KV)은 Cloudflare 유지, Vercel 캐시 MISS 때만 뒤로.

2. 최종 아키텍처

사용자(서울) ~7ms RTT ▼ Vercel Edge (ICN/서울) ── 캐시 HIT → 즉시 ~13ms ◀── 99% 여기서 끝 │ (MISS 일 때만, 함수 리전 icn1 고정) ▼ Cloudflare Worker (api.zihado.com, 도쿄) │ 엣지 캐시(Cache API) → KV → D1 + 월·지역 사전집계 테이블 ▼ 국토교통부 MOLIT API (백필 큐 / cron) 캐시 유지: ① cron 2개(코어 0·30분 / 지역 15·45분) ② SWR 7일 ③ 수동 트리거

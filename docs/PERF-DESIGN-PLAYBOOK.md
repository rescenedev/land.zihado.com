# 극한 성능 설계 플레이북

"로딩이 안 보이는" 수준(p50 ~10ms대, 거의 모든 요청 ≤50ms)을 **처음부터** 목표로 할 때의 설계 원칙. Landman 작업에서 도출.

> 한 줄 요약: **응답시간 = 네트워크 거리 + 서버 처리 + 페이로드 전송.** 셋 다 설계 단계에서 잡아야 하고, 가장 자주 간과되는 건 "네트워크 거리(어느 colo에서 서빙되나)"다.

---

## 0. 가장 먼저 — 응답시간을 3개로 분해하고 각각 목표를 정한다

```
총 응답시간 = ① 네트워크 RTT (사용자↔서빙 colo)
            + ② 서버 처리 (라우팅+쿼리+직렬화)
            + ③ 페이로드 전송 (크기/압축)
```

- ①이 보통 지배적이고 코드로 못 줄인다 → **인프라/위치 선택의 문제**. 먼저 본다.
- 50ms 목표라면: 사용자가 가까운 colo(RTT ≤15ms) + 서버 ≤10ms + 작은 압축 페이로드.
- **"빛 속도 한계"라고 단정하기 전에 provider별 colo를 실측하라.** 같은 회선에서도 provider마다 붙는 colo가 다르다(아래 1번).

---

## 1. 네트워크 거리 — provider/엣지 선택을 코드보다 먼저

가장 비싼 실수: 서버를 ms 단위로 튜닝했는데 트래픽이 엉뚱한 대륙 colo로 빠지는 것.

- **타깃 사용자 지역에서 각 provider의 실제 서빙 colo를 측정한다.** `curl`로 `cf-ray`/`x-vercel-id`/`fly-region` 헤더, `ping`으로 RTT.
- 같은 한국 회선이라도: Cloudflare 고객 zone → 도쿄(NRT, 37ms), Vercel → 서울(ICN, 7ms)로 갈렸음. 플랜 업그레이드로도 안 바뀜(BGP 피어링).
- **결론을 실측 없이 내리지 말 것.** "Pro 사면 ICN 된다" 같은 추측으로 비용 쓰지 말고, 먼저 무료/임시 배포로 colo를 찍어본다.
- 엣지에서 서빙할 데이터는 **사용자 근처 엣지에 두고**, 원본 DB는 멀어도 됨(캐시로 가림).

---

## 2. 캐시 계층 — "거의 항상 HIT"가 기본값이 되도록

목표는 캐시를 "있으면 좋은 것"이 아니라 **모든 읽기의 기본 경로**로 만드는 것.

- **계층화**: 브라우저 메모리 → 엣지 캐시(사용자 colo) → KV/응답캐시 → DB.
  - 엣지 캐시 HIT은 함수/워커조차 안 거치게(서버 처리 0에 수렴).
- **SWR(stale-while-revalidate)를 길게**: 만료돼도 stale 즉시 서빙 + 백그라운드 갱신 → 사용자는 절대 재계산을 기다리지 않음. 최초 1회만 cold.
- **inflight 중복제거**: cold miss 시 동시 요청이 같은 연산을 N번 돌리지 않게.
- **캐시 키 = 전체 URL.** 워밍/프리페치 URL을 클라이언트가 실제 보내는 것과 **파라미터 순서·값까지 바이트 단위로 일치**시켜야 같은 키. (가장 흔한 "워밍했는데 왜 MISS?" 원인)
- 캐시 TTL은 데이터 변동성에 맞춰: 불변(과거)=장기, 당월=짧게+SWR.

---

## 3. 데이터/쿼리 — 읽기 시점에 계산하지 말고 미리 만들어둔다

- **사전집계(materialized aggregate)**: overview/통계처럼 집계 종류가 고정이면, 읽기마다 raw를 GROUP BY 하지 말고 ingest 시점에 집계 테이블을 갱신. (130만행 스캔 → 1만행 룩업, 485ms → 1.3ms)
- **실시간 집계 금지**: 통계 백분위수처럼 무거운 계산은 cron으로 미리 계산·캐시. 사용자 요청 경로에 무거운 연산을 두지 않는다.
- **쿼리는 인덱스 시크가 되게**: 복합 인덱스 설계 + `ANALYZE`로 플래너 통계 최신화(통계 없으면 풀스캔 함). `EXPLAIN QUERY PLAN`으로 rows_read 확인이 습관.
- **N+1 / 직렬 왕복 제거**: 교차리전 DB 호출은 왕복이 비싸다 → 여러 쿼리를 한 배치로(3왕복 → 1).
- **스캔 범위를 키로 좁혀라**: 흔한 값(예: 전국 공통 단지명)으로 IN 하면 전국 스캔. 가능한 식별자(지역코드 등)를 함께 걸어 정밀 인덱스 사용.

---

## 4. 페이로드 — 작게, 압축되게

- 응답에서 불필요 필드 제거(예: trend `{month,avg}[]` → `number[]`, 31KB→21KB).
- 압축(br/gzip) 전제로 설계하고 **압축 켠 상태로 측정**(안 켜면 비관적 수치에 속음).
- 큰 목록은 limit/페이지네이션, 클라이언트가 실제 쓰는 만큼만.

---

## 5. 사전 워밍 — bounded 키는 전부 미리 굽는다

- **키 공간을 bounded/unbounded로 분류**:
  - bounded(메뉴·시도·최근 N일·지역코드): cron으로 **전부 사전 워밍** → 첫 진입도 HIT.
  - unbounded(임의 검색어·좌표별 상세): 사전 불가 → SWR로 "최초 1회만 cold, 이후 7일 HIT" 보장.
- **워밍은 사용자가 닿는 엣지 캐시를 데워야** 의미 있음(원본만 데우면 엣지는 여전히 MISS). 워밍 fetch가 엣지를 통과하게.
- **실행 한도를 의식**: 워커 subrequest 한도/함수 실행시간 때문에 한 번에 다 못 데우면 cron을 쪼개 각자 한도 내 완주하게.
- 우선순위 순서로 워밍(중간에 끊겨도 고가치 키부터).

---

## 6. 측정 — 처음부터 자동화, 증거로만 말한다

- **엔드포인트 레지스트리 + 1000회 측정 스크립트**를 초기에 만든다(회귀 감지).
- **연결 재사용/HTTP2 조건으로** 측정(실브라우저와 동일). ab처럼 keep-alive 못 하는 도구는 수치가 과장됨 — 도구 특성 이해.
- p50뿐 아니라 **p95/p99/max**까지. 꼬리는 표본 수에 민감(4천 요청이 1천보다 꼬리 김) — 조건 통일.
- 다중 도구 교차검증(k6/wrk/hey/oha/bombardier)으로 도구 편향 배제.
- **서버 처리와 네트워크를 분리 측정**: `time_connect`(1 RTT) vs `time_starttransfer`(TTFB) vs 전체. 어디서 새는지 봐야 옳게 고친다.

---

## 7. 안티패턴 체크리스트

- [ ] 서버만 튜닝하고 colo 라우팅을 안 봤다 (가장 큰 낭비)
- [ ] CDN/엣지를 앞단에 또 프록시해서 먼 colo를 경유시킴 (CF→Vercel→CF 같은 구조)
- [ ] 워밍 URL이 클라이언트 URL과 파라미터 순서가 달라 캐시 키 불일치
- [ ] 읽기 경로에서 매번 무거운 집계를 실시간 계산
- [ ] 압축 안 켜고 측정해서 페이로드가 느리다고 오판
- [ ] SWR 없이 짧은 TTL → 만료마다 사용자가 재계산 대기
- [ ] 한 번에 다 워밍하려다 실행 한도 초과로 뒤쪽이 안 데워짐

---

## 우선순위 (이 순서로 잡아라)

1. 타깃 지역 ↔ provider colo **실측** → 가장 가까운 엣지 선택
2. 엣지 캐시 + SWR을 읽기의 기본 경로로
3. 데이터는 사전집계, 읽기 경로에서 계산 금지
4. bounded 키 전부 사전 워밍(엣지를 데우게)
5. 페이로드 축소 + 압축
6. 1000회·다중도구 측정 자동화로 회귀 감시
